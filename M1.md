# M1 Execution Blueprint: MedTrace AI Knowledge Graph & Engines

## 1. M1 Complete Vision

As the Architecture Lead and Knowledge Graph Owner, Member 1 is the foundational pillar of the MedTrace AI platform. You are building the deterministic truth engine that powers the generative AI agents.

**What M1 is Building:** A living Digital Twin of a medical device's lifecycle within Neo4j, structured to map requirements, tests, risks, complaints, and CAPAs. You are also building the analytical engines (Audit Readiness and Ripple Propagation) that interrogate this graph.

**Why it is Required:** LLMs hallucinate; graphs do not. To simulate an adversarial FDA/MDR auditor, the system requires exact, deterministic relationships between firmware versions, verification evidence, and regulatory requirements.

**Problem Solved:** Regulatory teams rely on disconnected PDFs and spreadsheets, making it impossible to see when a change in one system (e.g., firmware) invalidates compliance evidence in another (e.g., test reports). M1's graph makes these hidden gaps immediately visible.

**Final Outcome:** A highly optimized Neo4j database exposed via a Python API layer (Cypher query library) that feeds exact graph neighborhoods and readiness scores to the LangGraph orchestration layer.

**Success Criteria & KPIs:**

* **Query Latency:** Graph neighborhood retrieval (3-hop depth) in under 150ms.
* **Traceability Coverage:** 100% of disconnected nodes flagged.
* **Engine Accuracy:** Audit Readiness Engine accurately calculates risk states matching manual ISO 13485 auditor assessments.
* **Resilience:** Zero orphan nodes allowed in the synthetic or real datasets.

| **Traceability** | Bidirectional traversal capabilities to support FDA 21 CFR 820.30 Design Control matrices. |

## 3. Generic Device Data Model

To support any medical device—from a Pulse Oximeter to an Insulin Pump or a Software as a Medical Device (SaMD)—the graph schema must use an abstraction layer. Hardcoding device-specific attributes as node labels is strictly prohibited.

### Device-Independent Schema

* **Abstract Labels:** `DeviceFamily`, `DeviceModel`, `System`, `Subsystem`, `Component`.
* **Regulatory Labels:** `Requirement`, `Specification`, `TestProtocol`, `Test`, `Evidence`, `Risk`, `Complaint`, `CAPA`.


* **Software Labels:** `SoftwareVersion`, `FirmwareVersion`, `SBOM_Component`, `CVE`.



### Plug-and-Play Onboarding Framework

When adding a new device (e.g., Insulin Pump), the core schema remains untouched. The onboarding process simply injects standard nodes with device-specific properties.

| Entity Type | Base Label | Extensibility Implementation |
| --- | --- | --- |
| **The Product** | `:Device` | Properties: `device_class="Class III"`, `nomenclature_code="GMDN-12345"`.

 |
| **Physical Parts** | `:Component` | Properties: `part_type="Motor"`, `supplier_id="V-992"`. |
| **Software** | `:SoftwareVersion` | Linked via `[:HAS_VERSION]` to specific `:Device` nodes.

 |
| **Compliance** | `:Requirement` | Properties: `standard="IEC_60601-1-8"`, `category="Alarm_Systems"`.

 |

**Future Device Addition Example:** To add an ECG Monitor, create a `:Device {name: "ECG-Pro"}` node and attach `:Component {name: "Lead Wire"}` nodes. The standard `:Requirement` and `:Risk` nodes seamlessly link to these new components using standard `[:AFFECTS]` or `[:MITIGATED_BY]` edges.

## 6. Audit Readiness Engine

The Audit Readiness Engine dynamically scores a device's compliance health in real-time. It evaluates the deterministic completeness of the traceability matrix.

### Scoring Logic & Formula

The score is calculated per Requirement Node and rolled up to the Device level.

$$S_R = (W_{E} \cdot E_X) + (W_{F} \cdot F_R) + (W_{A} \cdot A_C) - (W_{C} \cdot C_P)$$

**Variable Definitions:**

* $S_R$: Requirement Readiness Score (0-100).


* $W$: Configurable weights (e.g., $W_E = 40$, $W_F = 30$, $W_A = 30$, $W_C = 20$).
* $E_X$: Evidence Exists Boolean (1 if linked test exists, 0 if missing).


* $F_R$: Evidence Freshness (1 if test firmware matches current device firmware, 0 if stale).


* $A_C$: Acceptance Criteria Met (1 if LLM agent validates criteria, 0 if failed).


* $C_P$: Open CAPA Penalty (1 if node has an active CAPA, 0 if clear).



### Neo4j Cypher Execution (Data Flow)

1. **Start:** Define target Device and current Firmware Version.
2. **Traverse:** Find all Requirements linked to that Device.
3. **Evaluate:** For each Requirement, traverse to Tests, check CAPAs, and extract version metadata.

```cypher
MATCH (d:Device {id: $device_id})-[:HAS_VERSION]->(current_fw:SoftwareVersion)
MATCH (d)-[:CONTAINS]->(req:Requirement)
OPTIONAL MATCH (req)-[:VERIFIED_BY]->(test:Test)
OPTIONAL MATCH (req)<-[:ADDRESSES]-(capa:CAPA {status: 'Open'})
RETURN 
  req.id, 
  CASE WHEN test IS NOT NULL THEN 1 ELSE 0 END AS evidence_exists,
  CASE WHEN test.firmware_tested = current_fw.version THEN 1 ELSE 0 END AS evidence_fresh,
  count(capa) AS open_capa_count

```

## 7. Ripple Propagation Engine

When a change occurs (e.g., a component swap or firmware update), the Ripple Engine calculates the exact regulatory impact radius.

### Graph Traversal Strategy

You must use variable-length path matching to traverse the traceability matrix from the point of origin down to the validation evidence.

**Scenario:** Firmware v3.4 is updated to v3.5.

1. **Identify Change:** `(SoftwareVersion {version: "v3.5"})`
2. **Find Affected Requirements:** Traverse `[:AFFECTS]` edges.


3. **Find Stale Tests:** Traverse from Requirements via `[:VERIFIED_BY]` to Test nodes. If `Test.firmware_tested != "v3.5"`, flag as STALE.


4. **Find Associated Risks:** Traverse `[:MITIGATED_BY]` to ensure Risk mitigations are still valid.



### Performance Considerations

Limit variable depth to `*1..5` to prevent exhaustive graph scans, as traceability matrices rarely exceed 5 hops from input to output.

## 8. Regulatory Time Machine Architecture

The Regulatory Time Machine answers the fundamental auditor question: *"What did you know and when did you know it?"*.

### Implementation Strategy: Event-Sourcing Graph

Rather than modifying nodes in place, M1 must design an immutable graph layer.

* **Graph Versioning:** Every Node and Edge receives `valid_from_date` and `valid_to_date` properties.
* **Current State:** Defined where `valid_to_date IS NULL`.
* **Historical Queries:** To reconstruct the graph from 8 months ago, apply a global timestamp filter to the traversal.
* **PostgreSQL Role:** While Neo4j handles the structural traversal, PostgreSQL (managed by M5) stores serialized JSON snapshots of critical milestones (e.g., FDA submission dates) for instant retrieval.



```cypher
MATCH (d:Device {id: $device_id})-[r:CONTAINS]->(req:Requirement)
WHERE r.valid_from <= $target_date 
  AND (r.valid_to IS NULL OR r.valid_to > $target_date)
RETURN req

```

## 12. Final Execution Directives for M1

1. **Do not over-engineer the database.** Stick to Neo4j AuraDB's free tier capabilities. Keep the schema flat and rely on edge relationships for complexity.


2. **Focus on the Gotcha.** The entire pitch relies on AuditShadow catching the firmware mismatch. Ensure the `firmware_tested` property on the `Test` node and the `version` property on the `SoftwareVersion` node are perfectly formatted for strict equality comparison.


3. **Strictly enforce Contextual Integrity.** M2's LangGraph agents will hallucinate if M1's Cypher queries return fuzzy graph boundaries. Use precise `MATCH` paths, not open-ended `[*]` traversals, when serving data to the agents.
