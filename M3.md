You are translating the complex, deterministic knowledge graph (M1) and the multi-agent LLM reasoning (M2) into intuitive, high-fidelity visual insights.

* **User Adoption KPIs:** Zero learning curve; interface must feel immediately familiar to users of modern observability tools (e.g., Datadog, Palantir).

## 2. Complete Frontend Architecture

The frontend is a single-page application (SPA) experience built on Streamlit, leveraging modern state management and custom components for complex visualizations.

### Presentation Layer

* **Framework:** Streamlit (latest version).
* **Layout:** Wide mode by default (`st.set_page_config(layout="wide")`). Tabbed primary navigation.
* **Styling:** Custom CSS injected via `st.markdown(unsafe_allow_html=True)` for enterprise-grade spacing, typography, and dark mode optimizations.

### Visualization Layer

* **Graph Engine:** `streamlit-agraph` (React-based network graph) or `pyvis` (HTML iframe) for the Digital Twin.
* **Chart Engine:** Plotly Express / Plotly Graph Objects for interactive risk heatmaps, Sankey diagrams, and trend timelines.

### API Layer

* **REST Consumer:** Python `requests` library wrapped in `@st.cache_data` for static fetches.
* **WebSocket/SSE Consumer:** `httpx` or standard `requests` with `stream=True` to consume FastAPI Server-Sent Events (SSE) for live agent typing.

### State Management Layer

* **Core:** `st.session_state` is strictly typed and modularized.
* **Keys:** `current_device`, `audit_scope`, `graph_data`, `agent_stream_log`, `time_machine_date`.

### Streaming Layer

* Uses `st.write_stream()` combined with Python generators yielding chunks from the M5 FastAPI endpoints.

### User Personalization Layer

* Role-based toggle in the sidebar simulating different user perspectives (e.g., "View as FDA Auditor" vs. "View as Quality Manager").

## 6. Digital Twin Interface (MOST IMPORTANT)

This is a dynamic, interactive force-directed graph.

### Graph Design

* **Library:** `streamlit-agraph`.
* **Physics:** Barnes-Hut simulation enabled. Gravity tuned so the Device node is central, Requirements radiate outward, Tests and Evidence form the outer shell.

### Node Specifications

| Node Type | Shape | Icon / Label | Color Logic |
| --- | --- | --- | --- |
| **Device** | Hexagon | 💻 Device | Always Blue (#1E88E5) |
| **Requirement** | Box | 📋 Req | Score based: Green/Amber/Red |
| **Test** | Ellipse | 🧪 Test | Status based: Valid (Green), Stale (Red) |
| **Evidence** | Document | 📄 Doc | Confidence > 90% (Green) else (Amber) |
| **Risk** | Triangle | ⚠️ FMEA | Severity High (Red), Low (Yellow) |
| **CAPA** | Diamond | 🛑 CAPA | Open (Red), Closed (Green) |

### Interactions

* **Click Node:** Sidebar slides open (or columns adjust) to reveal the `Node Inspector`. Shows exact JSON metadata from M1.
* **Hover Node:** Fast tooltip showing `id`, `name`, and `audit_score`.
* **Filters (Top Bar):** "Hide Compliant Nodes", "Show Only Stale Firmware Links", "Isolate Path to Risk".

## 13. M3 ↔ M1 Integration Architecture (Neo4j)

**How M3 consumes M1 Data:**
M3 calls M5 (FastAPI), which acts as a proxy to M1.

* **Endpoint:** `GET /api/graph/twin/{device_id}`
* **Expected Payload:** ```json
{
"nodes": [{"id": "REQ-1", "label": "Requirement", "color": "green", "score": 95, "title": "Metadata..."}],
"edges": [{"source": "REQ-1", "target": "TEST-1", "label": "VERIFIED_BY"}]
}
```

```


* **Rendering Flow:** M3 parses this exact structure directly into `agraph(nodes=parsed_nodes, edges=parsed_edges)`.

## 14. M3 ↔ M2 Integration Architecture (Agents)

**How M3 consumes M2 Streaming Data:**
M3 connects to the Server-Sent Events (SSE) endpoint exposed by M5 for the LangGraph agents.

* **UI Implementation:**
```python
import requests
def stream_agent_output(payload):
    with requests.post("http://backend/api/agents/investigate", json=payload, stream=True) as r:
        for line in r.iter_lines():
            if line:
                yield process_sse_line(line)

st.write_stream(stream_agent_output({"complaint": user_input}))

```


* **Progress Tracking:** M2 yields state changes (e.g., `{"agent": "Risk", "status": "working"}`). M3 updates an `st.status()` or custom stepper component dynamically.

## 15. M3 ↔ M5 Integration Architecture (FastAPI Backend)

* **Base URL Configured via `.env`:** `BACKEND_URL=http://localhost:8000`
* **Error Handling:** Every API call must be wrapped in a `try/except` block. If M5 is down, M3 must render a stylized `st.error("Command Center disconnected from Intelligence Core.")`, not a raw Python traceback.
* **Authentication:** For the hackathon demo, mock a session token in `st.session_state.auth_token` and pass it in the Headers of all requests to M5.
