# M4 Execution Blueprint: The Regulatory Brain & Evidence Layer

## 1. Complete M4 Vision

**What exactly M4 builds:** Member 4 builds the "Regulatory Brain and Evidence Layer" of MedTrace AI. This encompasses the end-to-end data pipelines, real and synthetic document processing systems, the ChromaDB vector storage, and the high-precision Retrieval-Augmented Generation (RAG) architecture.

**Why M4 exists:** LLMs hallucinate, and in medical device compliance, a hallucination is a 483 warning letter or a patient safety risk. M4 exists to ground the generative agents (M2) in unshakeable, cryptographically traceable, and regulatory-compliant truth.

**How M4 contributes to MedTrace AI:** While M1 builds the relational map (Neo4j), M4 supplies the semantic depth (ChromaDB). When the AuditShadow agent asks, "Does this test satisfy the requirement?", M4 retrieves the exact paragraph from the test report, including its firmware version and page number.

**Business & Regulatory Value:**
Eliminates manual document hunting. Reduces the time to perform root-cause analysis and gap assessments from weeks to seconds by providing mathematically precise evidence retrieval with 100% traceable provenance.

**Success Metrics & KPIs:**

* **Retrieval Latency:** < 200ms per semantic search.
* **Context Precision:** > 95% (Top-3 retrieved chunks contain the exact answer).
* **Provenance Accuracy:** 100% of retrieved chunks contain valid `source_doc`, `page`, and `firmware_version` metadata.
* **Hallucination Rate:** 0% (forced by strict RAG context boundaries).

## 3. Generic Medical Device Data Framework

The architecture MUST NOT be hardcoded to a Pulse Oximeter. It uses a **Device Abstraction Ontology**.

### Device Ontology

* **`DeviceFamily`**: E.g., Infusion Pump, Wearable, ECG Monitor.
* **`DeviceModel`**: E.g., InfusionPro-X1.
* **`Subsystem`**: E.g., Fluid Delivery, Power Management, Display.
* **`Component`**: E.g., Stepper Motor, Li-Ion Battery.
* **`SoftwareComponent`**: E.g., MotorController.c, RTOS.

### Device Metadata Standards (JSON Schema)

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "Universal Device Manifest",
  "properties": {
    "device_class": { "type": "string", "enum": ["Class I", "Class II", "Class III"] },
    "regulatory_pathways": { "type": "array", "items": { "type": "string" } },
    "components": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "type": { "type": "string", "enum": ["Hardware", "Software", "Mechanical", "Biocompatible"] },
          "part_number": { "type": "string" }
        }
      }
    }
  }
}

```

**Onboarding Future Devices:** To add an "Insulin Pump," a user uploads a Device Manifest JSON. The system dynamically generates the missing synthetic documentation and populates the vector store with Pump-specific terminology without changing a single line of M4 Python code.

## 4. Complete Document Architecture

Every document type has a specific processing, chunking, and metadata strategy.

| Document Category | Purpose & Structure | Metadata Injected | Retrieval Strategy |
| --- | --- | --- | --- |
| **Design Inputs (URS/PRD)** | Define user/system needs. List-based. | `req_id`, `version`, `category` | Exact match on `req_id` + semantic search on requirement text. |
| **Design Outputs (Specs)** | Technical specs. Tabular/Text. | `spec_id`, `linked_req`, `component` | Hybrid search filtering by component. |
| **Verification & Validation** | Test protocols and results. | `test_id`, `firmware_version`, `result` | **Critical:** Strict exact match filter on `firmware_version`. |
| **Risk Management (FMEA)** | Hazards and mitigations. Tabular. | `hazard_id`, `rpn_score`, `severity` | Tabular data converted to Markdown text blocks before embedding. |
| **Complaint Records** | Post-market incidents. Unstructured. | `complaint_id`, `date`, `device_id` | Dense vector search to find similar historical symptoms. |
| **CAPA & NCR** | Corrective actions. Structured narrative. | `capa_id`, `root_cause_category` | Semantic search on root cause analysis text. |
| **Cybersecurity (SBOM/CVE)** | Security posture. JSON/XML. | `cve_id`, `cvss_score`, `library` | Parsed as JSON, embedded as text summaries. |
| **Regulatory Standards** | ISO 13485, Part 820. Hierarchical. | `clause`, `standard`, `year` | Hierarchical chunking (by section header). |

## 5. Real Data + Synthetic Data Strategy

### Hybrid Architecture

1. **Real Data Pipeline:** Processes actual user manuals, generic regulatory standards, and open-source CVE databases. Uses `PyMuPDF` for PDF parsing and `unstructured` for format normalization.
2. **Synthetic Data Pipeline:** Uses GPT-4o structured outputs and `Faker` to generate the proprietary Design History File (DHF) to avoid IP leakage during the hackathon.

**What is generated (Synthetic):**

* Test Reports, FMEA matrices, Complaint logs, CAPA records.
* *Crucial Demo Asset:* The "Gotcha" Test Report (linked to old firmware v2.1 while current is v3.4).

**What is NEVER generated (Real):**

* Regulatory standards text (e.g., MDR Annex I).
* Standard definitions (e.g., ISO 14971 Severity Matrices).

**Validation Process:**
A deterministic script runs post-generation to ensure the synthetic text contains the exact string IDs (e.g., `TR-2024-018`) required by M1's graph edges.

## 6. Knowledge Extraction Architecture

**Pipeline Flow:**
`Raw PDF` $\rightarrow$ `LayoutParser` $\rightarrow$ `Markdown Text` $\rightarrow$ `LLM Extraction Node` $\rightarrow$ `Structured JSON + Chunks`

**Extraction Node (Using GPT-4o-mini):**

* **Input:** 1000-token Markdown chunk.
* **Output:** JSON matching a strict Pydantic schema mapping Entities (Devices, Tests, Requirements) and Relationships (`VERIFIES`, `MITIGATES`).
* **Graph Delivery:** JSON payload is pushed to an SQS queue or direct API endpoint owned by M1.
* **Vector Delivery:** The original text chunk, appended with the extracted entities in its metadata, is pushed to ChromaDB.

## 7. RAG Architecture (MOST IMPORTANT)

A production-grade, hallucination-resistant Advanced RAG pipeline.

### Ingestion Pipeline

1. **Chunking Strategy:** `MarkdownHeaderTextSplitter` (splits by `##` and `###`) followed by `RecursiveCharacterTextSplitter` (chunk_size: 800, overlap: 100). Preserves document hierarchy.
2. **Embedding Strategy:** OpenAI `text-embedding-3-small` (1536 dimensions) for speed and cost-efficiency.

### Retrieval Strategy

1. **Query Rewriting:** LLM rewrites the agent's query to expand acronyms (e.g., "thermal runaway" $\rightarrow$ "battery overheat, thermal runaway, ISO 60601-1 battery safety").
2. **Hybrid Search:** ChromaDB vector similarity + exact Metadata filtering (e.g., `where={"firmware_tested": "v3.4"}`).
3. **Re-Ranking:** `CrossEncoder` (`ms-marco-MiniLM-L-6-v2`) re-ranks the top 15 results down to the top 5 most contextually relevant chunks.

### Context Assembly & Provenance

The final string passed to the M2 Agent includes strict provenance wrappers:

```xml
<document>
  <metadata>
    <source>test_report_battery_v2.1.pdf</source>
    <firmware>v2.1</firmware>
    <page>4</page>
  </metadata>
  <content>
    The battery pack was subjected to thermal cycling between -20C and +60C. 
    Result: PASS. Max temperature observed: 48C.
  </content>
</document>

```

## 10. Provenance Architecture

Every piece of evidence must be bulletproof.

**Source Tracking:**
ChromaDB IDs are generated via deterministic hashes of the text chunk: `hashlib.sha256(text.encode()).hexdigest()`.

**Citation Formatting:**
M2 agents are prompted to append ``. M4 provides a reverse-lookup API (`/api/data/provenance/{chunk_id}`) that the frontend (M3) uses to expand `chunk_id` into a hoverable tooltip containing the original PDF name, page, and exact text block.

## 12. M4 ↔ M1 Integration Architecture

**Data feeding the Knowledge Graph:**
M4 processes documents, extracts metadata, and calls M1 to build the graph.

1. M4 parses `test_report_v3.4.pdf`.
2. M4 extracts `TestID: TR-042`, `ReqID: REQ-011`, `Firmware: v3.4`.
3. M4 POSTs to M1 API: `/api/graph/edge/verify`. Payload: `{"source": "REQ-011", "target": "TR-042", "properties": {"firmware": "v3.4"}}`.
4. M4 saves the actual text into ChromaDB, keeping the Chroma `chunk_id` stored inside the M1 Neo4j node as `vector_ref`.

## 21. Future-Proof Multi-Device Architecture

To ensure the architecture is not locked to a Pulse Oximeter, M4 implements a **Manifest-Driven Ingestion** pattern.

**How it works:**

1. **Device Definition:** A new device is introduced via a configuration file: `manifests/insulin_pump.yaml`.
2. **Taxonomy Loading:** M4 parses the YAML to understand the specific taxonomy (e.g., `Fluidic_Subsystem`, `Stepper_Motor`).
3. **Dynamic Generation:** The `synthetic/generate_dhf.py` script reads the YAML. Instead of prompting the LLM for "Pulse Oximeter LEDs", it prompts for "Insulin Pump Fluidic Subsystems".
4. **Retrieval Adaptation:** The metadata schema allows arbitrary `device_family` tags. The retriever dynamically filters based on the `current_device` state passed by M3 (Frontend).

**Result:** MedTrace AI can switch from auditing an ECG Monitor to auditing an Implantable Pacemaker instantly, with the underlying data pipelines dynamically organizing the specific evidence for each.
