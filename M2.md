# M2 Execution Blueprint: MedTrace AI Intelligence Layer & Multi-Agent OS

## 1. M2 Complete Vision

**What M2 Builds:** Member 2 is architecting the "Intelligence Core" or the AI Operating System of MedTrace AI. You are building a stateful, multi-agent orchestration layer using LangGraph that acts as the cognitive engine traversing the Neo4j knowledge graph and ChromaDB vector store.

**Why M2 Exists:** A database simply stores relationships; it does not reason. M2 exists to apply regulatory reasoning to deterministic data. You bridge the gap between static compliance records and active, adversarial regulatory interrogation.

**How M2 Contributes:** M2 is the "brain" that drives the two core workflows: the proactive AuditShadow simulated inspection, and the reactive 5-agent Complaint-to-CAPA investigation pipeline.

**Business Problem Solved:** Eliminates the human cognitive overload required to manually cross-reference disconnected QMS (Quality Management System) tools. Turns weeks of CAPA investigation into a 3-minute, fully cited autonomous workflow.

**Expected Final Outcome:** A highly resilient, streaming multi-agent backend that passes structured, hallucination-free JSON payloads and cited prose back to the Streamlit frontend via FastAPI.

**Success Metrics & KPIs:**

* **Pipeline Latency:** Complete Complaint-to-CAPA generation in under 180 seconds.
* **Audit Accuracy:** 100% detection rate of deliberately planted graph anomalies (e.g., stale firmware evidence).
* **Hallucination Rate:** 0% (enforced by strict Pydantic parsing and M1/M4 graph constraints).
* **Citation Density:** Every regulatory assertion must contain exactly one `[Source, Confidence]` tag.

## 2. Complete Multi-Agent Architecture

### Agent Layer

The Agent Layer utilizes a **Supervisor-Worker routing model** managed by LangGraph.

* **Supervisor:** A lightweight routing node that reads the current `AgentState` and determines the next node to execute based on conditional logic (e.g., if Risk is High, route to CAPA Agent; if Risk is Low, route to Report Agent).
* **Workers:** Specialized LLM nodes (GPT-4o) equipped with specific tools (API calls to M1/M4) and restricted scopes.

### Orchestration Layer (LangGraph)

* **State Management:** Driven by a global Pydantic `GraphState` object passed between nodes.
* **Parallel Execution:** Used during Evidence Gathering. Multiple document chunks are analyzed concurrently by spawned instances of the Evidence Agent.
* **Human-in-the-Loop:** Supported via LangGraph's `interrupt_before` capability. The graph pauses before the final CAPA Agent execution to allow a human user to review the Root Cause.

### Memory Layer

* **Short-Term Memory:** The `GraphState` object (JSON) exists for the duration of a single request. It holds the original complaint, extracted entities, and accumulated context.
* **Long-Term Memory:** LangGraph's built-in `MemorySaver` (backed by PostgreSQL/SQLite) stores checkpoints of the graph state at every step, allowing the Regulatory Time Machine to replay investigations.
* **Regulatory Memory:** Stored in the System Prompt definitions, mapping directly to ISO 13485/FDA 21 CFR Part 820 clauses.

## 3. Agent Architecture Design

### Agent 1: Complaint Intake Agent

| Attribute | Design |
| --- | --- |
| **Purpose** | Parse raw human text into structured entity data. |
| **Inputs** | Raw string (User Complaint). |
| **Outputs** | JSON: `{device_id, severity, symptom_codes, affected_component, timeline}` |
| **Internal Workflow** | Extract entities via LLM structured output -> Map to ISO 14971 severity scale. |
| **Prompt Strategy** | Strict extraction prompt. No reasoning allowed. "You are an entity extractor. Output valid JSON only." |
| **Integration** | Calls M1 `find_similar_complaints(symptom_codes)` to append historical context. |

### Agent 2: Root Cause Agent

| Attribute | Design |
| --- | --- |
| **Purpose** | Generate testable hypotheses for the failure based on intake data. |
| **Inputs** | Structured Complaint + Historical Similar Complaints. |
| **Outputs** | List of top 3 Root Cause Hypotheses with base probabilities. |
| **Internal Workflow** | Apply 5-Whys methodology to the affected component -> Formulate hypotheses. |
| **Prompt Strategy** | Chain-of-Thought (CoT). "Under ISO 13485 Section 8.5.2, analyze the following symptom..." |
| **Integration** | Calls M1 to get the 2-hop graph neighborhood of the affected component. |

### Agent 3: Evidence Agent

| Attribute | Design |
| --- | --- |
| **Purpose** | Validate or refute root cause hypotheses using actual device documentation. |
| **Inputs** | Root Cause Hypotheses. |
| **Outputs** | Evidence context block with citation metadata. |
| **Internal Workflow** | Query ChromaDB (M4) using hypotheses -> Filter by current firmware -> Rerank results. |
| **Prompt Strategy** | "Act as a forensic auditor. Prove or disprove the hypothesis using ONLY the provided text." |
| **Integration** | Calls M4 (RAG) vector search endpoints. |

### Agent 4: Risk Agent

| Attribute | Design |
| --- | --- |
| **Purpose** | Recalculate device risk profile based on the confirmed root cause. |
| **Inputs** | Confirmed Root Cause + Verified Evidence. |
| **Outputs** | Updated FMEA hazard scores, Regulatory Reportability Flag (e.g., MDR Art. 87). |
| **Internal Workflow** | Fetch existing FMEA node -> Assess if occurrence/severity probability increased -> Update score. |
| **Prompt Strategy** | "Calculate Risk Priority Number (RPN). Compare against ISO 14971 acceptability matrix." |
| **Integration** | Calls M1 `recalculate_risk(node_id, new_probability)`. |

### Agent 5: CAPA Agent

| Attribute | Design |
| --- | --- |
| **Purpose** | Draft the formal Corrective and Preventive Action plan. |
| **Inputs** | Root Cause + Risk Profile + Evidence. |
| **Outputs** | Markdown formatted CAPA document with `[Source, Confidence]` citations. |
| **Internal Workflow** | Compile data -> Draft Problem Statement -> Draft Containment -> Draft Corrective Action -> Draft Verification Plan. |
| **Prompt Strategy** | "You are a QA Director. Write a concise, FDA-compliant CAPA. Cite all claims." |
| **Integration** | Emits final result to Streamlit frontend (M5). |

### Agent 6: AuditShadow Agent

| Attribute | Design |
| --- | --- |
| **Purpose** | Act as an adversarial FDA/MDR inspector. **(Flagship Feature)** |
| **Inputs** | Device Scope, Submission Type (e.g., 510(k)). |
| **Outputs** | FDA 483 style Observation Punch-List. |
| **Internal Workflow** | Described comprehensively in Section 7. |
| **Prompt Strategy** | "You are a merciless FDA investigator. Your job is to find traceability gaps." |
| **Integration** | Queries M1 for full regulatory matrices. |

### Agent 7: Trace Decay Agent

| Attribute | Design |
| --- | --- |
| **Purpose** | Background watcher for system changes. |
| **Inputs** | Graph Mutation Event (e.g., New Firmware Commit). |
| **Outputs** | Stale Traceability Alert list. |
| **Internal Workflow** | Traverse graph from changed node -> Find mismatched edge properties (e.g., v3.4 vs v2.1). |

## 7. AuditShadow Architecture (FLAGSHIP)

AuditShadow does not just read documents; it acts as an adversarial state machine evaluating deterministic logic against semantic evidence.

### The AuditShadow Brain (Execution Chain)

1. **Scope Definition:** Receive `device_id` and target `standard` (e.g., ISO 13485).
2. **Graph Traversal (Deterministic):** Request M1 to provide all Requirements. For each Requirement, traverse to its linked Verification Test.
3. **Integrity Check 1 (Version Matrix):** Does `Test.firmware_version` exactly match `Device.current_firmware`? If NO -> **Generate Finding**.
4. **Integrity Check 2 (Semantic):** If versions match, extract `Requirement.acceptance_criteria` and `Test.result_text`. Pass to LLM.
5. **Reasoning Engine:** "Does the text in the Test Report actually satisfy the specific bounds of the Acceptance Criteria, or is it just vaguely related?" If NO -> **Generate Finding**.
6. **Integrity Check 3 (Open CAPA):** Does this Requirement have an open CAPA linked in the graph? If YES -> **Flag as High Risk**.

### AuditShadow Persona Prompt

```markdown
SYSTEM PROMPT: AUDIT_SHADOW_CORE

You are AuditShadow, a Senior FDA Investigator conducting a Part 820 inspection.
You are merciless, detail-oriented, and deeply suspicious of documentation.
You do not assist the manufacturer. You exist to find non-conformities.

Given the following Requirement and Evidence pair, assess compliance.
- Do NOT assume compliance because a document exists.
- Verify exact firmware versions match.
- Verify the acceptance criteria numbers are explicitly met in the test results.

If ANY gap exists, output a structured Observation in standard FDA 483 format:
[OBSERVATION] 
- Regulatory Reference: [e.g., 21 CFR 820.30(g)]
- Finding: [Detailed explanation of the gap]
- Risk Level: [Critical/Major/Minor]

If no gap exists, output exactly: {"status": "COMPLIANT"}

```

## 8. Regulatory Knowledge Framework

Instead of hardcoding rules, M2 uses a **Contextual System Prompt Injector**.

**Implementation:**
When a process starts, the user selects a Regulatory Framework via the frontend. The Orchestrator fetches the corresponding Markdown framework file (e.g., `iso_13485_rules.md`) and injects it into the `System Message` of the Agent.

**Supported Mappings:**

* **FDA 21 CFR 820:** Focuses on "Design History Files" and "Device Master Records".
* **ISO 13485:** Focuses on "QMS processes" and "Management Review".
* **EU MDR:** Focuses on "General Safety and Performance Requirements (GSPR)" and "Post-Market Clinical Follow-up (PMCF)".

## 9. Generic Device Intelligence

M2 must be **Device-Agnostic**. Agents cannot contain hardcoded logic for "Pulse Oximeter LEDs".

**Abstraction Strategy:**

1. Before any investigation, the Orchestrator executes a `get_device_context(device_id)` call to M1.
2. M1 returns a JSON topological map of the device (e.g., `Device -> System -> Subassembly -> Component`).
3. The LLM prompt dynamically receives this topology.
4. *Example:* Whether analyzing an Insulin Pump's "stepper motor" or a Ventilator's "pressure valve", the agent simply analyzes the `Component` node and its associated `[:MITIGATES]` edges.

## 10. Prompt Engineering Architecture

Every agent utilizes a strict 3-tier prompt architecture to prevent drift and hallucination.

| Prompt Type | Description | Location in LangGraph |
| --- | --- | --- |
| **System Persona** | Defines role, tone, and absolute constraints (e.g., "You are a QA Director. Never invent data."). | Base Node Initialization |
| **Task Prompt** | Specific instructions for the current state (e.g., "Analyze this specific complaint text: [TEXT]"). | Per-Invocation |
| **Self-Critique** | "Review your previous output. Did you include citations for every claim? If not, rewrite." | Validation Node |

## 13. M2 ↔ M4 Integration Architecture (ChromaDB)

M2 triggers M4 RAG capabilities via LangChain Tools.

**Retrieval Flow:**

1. Evidence Agent formulates a search query: `"Battery thermal runaway test protocols v3.4"`.
2. Calls M4 `retrieve_documents(query, filter={"firmware": "v3.4"})`.
3. M4 returns top-K chunks with `source` and `page_number` metadata.
4. Evidence Agent injects chunks into its context window.
5. **Provenance:** Agent appends `[Source: {metadata.source}, Page: {metadata.page}]` to its generated text.

## 14. M2 ↔ M5 Integration Architecture (FastAPI)

M2 must stream its LangGraph execution state back to the UI in real-time.

* **Mechanism:** Server-Sent Events (SSE) via FastAPI.
* **Event Handling:** M2 yields a dictionary at every LangGraph step: `{"agent": "Risk Agent", "status": "Calculating RPN...", "data": null}`.
* M5 forwards these WebSocket tokens to Streamlit to update the UI progress bars.

## 16. Failure & Recovery Architecture

| Failure Scenario | Fallback Logic (LangGraph Router) |
| --- | --- |
| **Neo4j Timeout (M1)** | Retry x3 with exponential backoff. If fail, return graceful error to UI. |
| **LLM Output Invalid JSON** | Trigger `ValidationNode`. Append error prompt: "Output was invalid JSON. Fix it." Loop max 3 times. |
| **Missing Evidence (M4)** | Evidence Agent returns `null`. Graph routes to CAPA Agent with instruction: "Draft CAPA based on missing documentation finding." |
